# Архитектура: единый движок доступности (Sheber Qonaq)

Статус: черновик на согласование · 20.06.2026

Цель: один движок бронирования под **суточный, понедельный и помесячный** режимы одновременно, без разделения бизнеса на «типы». Поддержать бронирование вперёд с резервом и предоплатой.

---

## 1. Главный принцип

**Доступность считается из диапазонов дат броней, а не из `unit.status`.**

Одна койка (Unit) держит много непересекающихся по датам броней. `unit.status` остаётся только операционным (свободна/грязь/ремонт/выведена), и НЕ является источником «занято/свободно» для календаря.

Следствие: помесячный гость — это просто очень длинная бронь на той же шкале. Суточный — короткая. Движку всё равно: он работает с интервалами `[check_in, check_out)`.

### Полуоткрытый интервал

Бронь занимает **ночи** `[check_in_date, check_out_date)`. День выезда свободен для нового заезда (отельный стандарт: гость уехал утром — новый заехал днём).

Две брони пересекаются ⟺ `a.check_in < b.check_out AND b.check_in < a.check_out`.
Заезд в день чужого выезда → НЕ пересечение (это правильно).

---

## 2. Единая модель тарифов

`rate_type ∈ {daily, weekly, monthly}` — это просто поле на той же `Stay` с интервалом `[check_in, check_out)`. Никаких отдельных «режимов» в коде.

`total_expected` уже считает по типу: daily×ночи, weekly×ceil(недель), monthly×календарные месяцы, смена×фикс. Менять не нужно.

Cottage-смены (`shift_type`) — внутрисуточная подсистема, оставляем ортогональной (см. §5). Основной движок дат её не трогает.

---

## 3. Жизненный цикл брони (статусы)

```
   создание                предоплата            заезд              выезд
  ─────────► reserved ───────────► confirmed ───────► active ───────► checked_out
                │                      │                 │
                └──── cancelled ◄───────┴──── no_show ◄───┘
                          expired (авто, по таймауту резерва)
```

| Статус        | Смысл                                   | Занимает даты? |
|---------------|-----------------------------------------|:--------------:|
| `reserved`    | Бронь создана, оплаты нет                |       ✅       |
| `confirmed`   | Внесена предоплата, бронь твёрдая        |       ✅       |
| `active`      | Гость фактически заселился                |       ✅       |
| `checked_out` | Выехал (история)                         | ❌ (только прошлое) |
| `cancelled`   | Отменено вручную                         |       ❌       |
| `no_show`     | Не явился                                |       ❌       |
| `expired`     | Резерв протух без предоплаты (авто)       |       ❌       |

**Блокирующее множество** `B = {reserved, confirmed, active}` — только эти занимают даты.

### Переходы (явные API-действия, каждый под guard)

- `POST /stays/` → создаёт `reserved`. **Walk-in** (заезд сегодня, оплата на месте) → можно создать сразу `active` (флаг в запросе) — РЕШЕНО: разрешаем быстрый путь.
- `POST /stays/{id}/confirm/` (+ запись предоплаты) → `confirmed` (см. §7а — порог предоплаты).
- `POST /stays/{id}/check-in/` → `active` (если `reserved`/`confirmed` и дата ≈ сегодня).
- `POST /stays/{id}/checkout/` → `checked_out` (как сейчас; ставит `unit.status='dirty'`).
- `POST /stays/{id}/cancel/` → `cancelled`.
- `POST /stays/{id}/no-show/` → `no_show`.
- `expired` — Celery-задача снимает резерв без предоплаты **через 12 часов** (РЕШЕНО: 12ч, не 24 — для суток долго).

Текущие данные: существующие `active`-брони остаются `active` и продолжают блокировать свой диапазон — ничего не ломается.

## 4. Защита от двойного бронирования

Два уровня.

**a) Application-level (работает везде, в т.ч. sqlite/тесты):**
В сериализаторе при создании/переходе — `SELECT FOR UPDATE` на Unit + проверка отсутствия пересекающейся брони в статусах `B`. Это обобщение текущего cottage-паттерна на произвольный диапазон.

**b) DB-level (PostgreSQL, золотой стандарт против гонок):**
Exclusion-констрейнт через GiST:

```sql
CREATE EXTENSION IF NOT EXISTS btree_gist;
ALTER TABLE stays_stay ADD CONSTRAINT no_overlap_per_unit
  EXCLUDE USING gist (
    unit_id WITH =,
    daterange(check_in_date, expected_check_out_date, '[)') WITH &&
  ) WHERE (shift_type = '' AND status IN ('reserved','confirmed','active'));
```

- `WHERE shift_type = ''` — cottage-смены исключены из дат-констрейнта (у них своя внутрисуточная проверка).
- Только Postgres. На sqlite (тесты) констрейнта нет → полагаемся на app-level + тесты.
- **Заменяет** текущий `unique_active_stay_per_unit`.

Миграция безопасна: сейчас «одна активная на юнит», пересечений в блокирующих статусах нет → констрейнт встанет чисто.

---

## 5. Cottage-смены (ортогонально)

Внутрисуточные смены (`day`/`night`/`full`) остаются как есть: своя overlap-проверка по `(unit, check_in_date, shift_type)`. Дат-движок и дат-констрейнт их не касаются (`shift_type != ''`). Будущая унификация в тайм-интервалы `[start_dt, end_dt)` возможна, но не сейчас — не нужна для текущей цели.

## 6. API доступности

```
GET /api/v1/availability/?from=2026-06-23&to=2026-06-26&unit_type=bed&property=<id>
```
Возвращает юниты, свободные на **весь** диапазон `[from, to)` (нет пересекающейся брони в статусах `B`), с ценой за ночь и итогом за период.

Запрос (Postgres):
```sql
SELECT u.* FROM properties_unit u
WHERE u.room_id IN (...property...) AND (u.unit_type = :type OR :type IS NULL)
AND NOT EXISTS (
  SELECT 1 FROM stays_stay s
  WHERE s.unit_id = u.id AND s.shift_type = ''
    AND s.status IN ('reserved','confirmed','active')
    AND daterange(s.check_in_date, s.expected_check_out_date, '[)')
        && daterange(:from, :to, '[)')
);
```

```
GET /api/v1/occupancy/calendar/?from=&to=&property=<id>
```
Возвращает по каждой ночи: занято/всего (для тепловой карты месяца).

## 7. Цены (база + сезонность)

Цены должны быть **редактируемыми и сезонными** (РЕШЕНО). Делаем в два слоя, по фазам.

**Фаза A — базовые тарифы (сначала).**
`base_rates` (JSON `{daily, weekly, monthly}`) по `unit_type` на уровне Property (РЕШЕНО: по типу юнита — койка/комната/апартамент стоят по-разному). По аналогии с уже рабочим `shift_rates`. Персонал может переопределить на конкретной брони (`rate_amount`).

**Фаза B — сезонные правила (потом, отдельный PR).**
Модель `RateRule`: `property`, `unit_type` (или конкретный unit), `date_from`, `date_to`, `daily/weekly/monthly` (или множитель), `priority`. При расчёте цены за ночь движок берёт правило с наибольшим приоритетом, чьё окно покрывает дату; иначе — `base_rates`. Так добавляются «высокий сезон», «новогодние», промо-периоды — без правки кода.

Котировка брони (`POST /stays/quote/`) проходит по каждой ночи диапазона и суммирует цену с учётом сезонных правил → показывает гостю/ресепшну итог до создания брони.

### 7а. Политика предоплаты (порог confirmed)

РЕШЕНО: предоплата процентом, ориентир **40–50%**. Рекомендация на старт — **флаговый процент** `deposit_percent` (по умолчанию 50%) в настройках объекта; `confirmed` требует `total_paid ≥ deposit_percent × total_expected`.

Тонкость «от скольки дней»: длину можно учесть тиры (напр. 1–2 ночи → 100% или 1 ночь; 3+ → 50%), но это правила, которые ты ещё не зафиксировал. Совет: стартуем с плоского настраиваемого %, тиры по длительности добавим правилом позже, когда обкатаешь на практике. Так не блокируем движок.

## 8. Риски и как снимаем

1. **`unit.status` больше не «занято».** Дашборд сейчас считает занятость по `unit.status='occupied'`. Надо переучить его считать из активных броней/диапазонов. Трогает рабочий дашборд → отдельный аккуратный шаг + тест.
2. **Exclusion-констрейнт только в Postgres.** Тесты на sqlite его не проверят → app-level проверка должна быть железная и покрыта тестами.
3. **Cottage + констрейнт** — снято через `WHERE shift_type=''`.
4. **Сложность стейт-машины** — каждый переход с guard'ами и тестами.
5. **Авто-expire резервов** (Celery) — опционально; добавляет инфру. Можно стартовать без него (резервы снимаются вручную), включить позже.
6. **Старые данные** — `active` остаётся `active`, мигрируется само.

## 9. Порядок сборки (по PR, каждый под зелёный CI)

1. **Статусы + стейт-машина**: добавить `reserved`/`confirmed`/`expired`, действия confirm/check-in/cancel/no-show + walk-in→active. Тесты переходов.
2. **Overlap-движок**: заменить `unique_active_stay_per_unit` → app-level проверка + Postgres exclusion. Тесты пересечений (заезд в день выезда разрешён, нахлёст запрещён).
3. **Базовые цены (Фаза A)** `base_rates` по unit_type + `POST /stays/quote/`. Тесты котировки.
4. **API доступности**: поиск свободных по датам + календарь занятости. Тесты.
5. **Фронт**: поиск по датам → свободные койки → резерв/предоплата (по макету), тепловая карта месяца.
6. **Дашборд**: занятость из броней, а не из `unit.status`.
7. **Авто-expire** резервов 12ч (Celery).
8. **Сезонные цены (Фаза B)** `RateRule` — отдельным заходом после обкатки базового движка.

## Решения (20.06.2026)

1. **Walk-in** → можно сразу `active` (быстрый путь на ресепшн).
2. **Авто-expire** резервов → авто через **12 часов** без предоплаты (Celery).
3. **Предоплата** → процент 40–50%; старт с флагового `deposit_percent` (по умолч. 50%), тиры по длительности — позже.
4. **Цены** → база по `unit_type` на Property (Фаза A) + сезонные правила `RateRule` (Фаза B). Редактируемые.
