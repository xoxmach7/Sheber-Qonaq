# Self-service регистрация клиентов + удаление поля "Тарифный план" из онбординга

Дата: 2026-07-08

## Контекст

Сейчас единственный способ завести нового клиента (хостел/гестхаус) в Sheber Qonaq — форма `/onboarding`, доступная только `SuperAdmin`. Он вручную вводит объект, комнаты и создаёт менеджера. В форме есть неиспользуемое поле "Тарифный план" (select: Бесплатный/Базовый/Профессиональный), которое ни на что не влияет — тарифы не реализованы.

Цель: снять с админа рутину — дать клиентам самим регистрироваться и заводить свой объект, при этом:
- защититься от злоупотреблений (создание кучи фейковых объектов одним человеком);
- ограничить бесплатный период 30 днями, после чего аккаунт переходит в read-only;
- подготовить FAQ-контент для внешнего лендинга, объясняющий процесс регистрации.

Админский `/onboarding` остаётся как есть и продолжает работать (для ручных случаев/помощи клиентам), из него только убирается поле "Тарифный план".

## Часть 1 — Убрать "Тарифный план" из онбординга

В [Onboarding/index.tsx](frontend/src/pages/Onboarding/index.tsx) удалить блок select "Тарифный план" (строки ~196–203) из шага "Объект". Состояние `plan` остаётся в коде, но фиксируется как константа `'free'` — отправляется в payload без пользовательского выбора. Поле `Organization.plan` на бэкенде не трогаем — оно уже есть и используется как задел под будущий биллинг.

## Часть 2 — Self-service регистрация

### Пользовательский поток

1. Клиент открывает публичную страницу `/signup` в этом же frontend (Sheber Qonaq).
2. Заполняет: email, пароль, название объекта, город, тип объекта (хостел/гестхаус).
3. Получает письмо со ссылкой подтверждения (через Resend).
4. Переходит по ссылке `/signup/confirm/:token` → аккаунт активируется: создаются Organization, Property, manager-User с ролью владельца, `trial_ends_at = now + 30 дней`.
5. Клиент логинится и дальше сам добавляет комнаты/юниты (используя существующий intra-app UI, а не форму онбординга целиком — комнаты создаются позже, не в момент регистрации, чтобы не дублировать многошаговый визард).
6. Максимум 20 юнитов (комнат/домиков) на организацию — валидируется на бэкенде при создании юнита, не только на фронте.
7. Одна регистрация на email — при повторной попытке с тем же email или совпадающим телефоном показываем ошибку "уже зарегистрировано, войдите или восстановите пароль".

### Ограничение триала

- Поле `trial_ends_at` на `Organization` (новое, `DateTimeField`, nullable — null означает "не триал", т.е. заведено вручную суперадмином без ограничения).
- Когда `trial_ends_at` в прошлом и нет активной оплаченной подписки (сейчас оплаты нет — значит всегда read-only после истечения), API блокирует create/update/delete для бронирований и связанных сущностей (units, guests и т.п. по вкусу, минимально — бронирования) с 403 и понятным сообщением ("Пробный период закончился, свяжитесь с нами для подключения подписки"). GET-запросы остаются разрешены.
- Реализация — DRF permission-класс, подключаемый к нужным ViewSet'ам, проверяющий `request.user.organization.trial_ends_at`.
- Баннер в шапке дашборда: если триал активен — "Осталось N дней бесплатного периода"; если истёк — "Пробный период закончился. Доступ только для просмотра. [Написать нам]".

### Backend

- Новая модель `SignupRequest` (app `organizations`): `email`, `org_name`, `city`, `booking_mode`, `password_hash`, `token` (uuid), `created_at`, `confirmed_at` (nullable).
- `POST /api/organizations/signup/` — валидирует уникальность email/названия, создаёт `SignupRequest`, отправляет письмо через Resend с ссылкой `{FRONTEND_URL}/signup/confirm/{token}`.
- `POST /api/organizations/signup/confirm/<token>/` — если `confirmed_at` пусто и токен не старше, например, 24 часов: создаёт `Organization` (`plan='free'`, `trial_ends_at=now+30d`), одну `Property`, `User` (manager/owner) с переданным паролем; помечает `confirmed_at`; возвращает токены авторизации (аналогично логину), чтобы клиент сразу попал в приложение.
- Rate limiting на signup endpoint (проверить, что throttle настроен — сейчас throttle классов нет вообще в проекте, см. memory аудита; для signup стоит добавить хотя бы базовый anon-throttle, чтобы не заспамили Resend).
- Настройки email: `EMAIL_BACKEND` через Resend (например, `django-anymail` с Resend backend, либо прямой HTTP-вызов Resend API). Требуется `RESEND_API_KEY` в Railway env — заводит пользователь до реализации.

### Frontend

- `/signup` — публичная страница с формой (email, пароль, название объекта, город, тип). После отправки — экран "проверьте почту".
- `/signup/confirm/:token` — вызывает confirm endpoint, при успехе логинит и редиректит на дашборд; при ошибке (истёкший/неверный токен) — понятное сообщение и ссылка "зарегистрироваться заново".
- Баннер триала в основном layout дашборда (виден только владельцу/менеджеру своей организации), использует `trial_ends_at` из ответа `/auth/me` или профиля организации.
- Валидация лимита 20 юнитов — на фронте при добавлении комнаты disable кнопки "Добавить" при достижении лимита + серверная ошибка как fallback.

### Что не входит в эту итерацию

- Реальная оплата/биллинг (только read-only-заглушка после триала).
- SMS-подтверждение телефона.
- Автоматические email-напоминания за N дней до конца триала (можно добавить позже).

## Часть 3 — FAQ для лендинга

Готовится отдельный markdown-файл `docs/faq-signup.md` с вопросами и ответами на русском: как зарегистрироваться, что входит в бесплатный период, лимиты (1 объект, до 20 юнитов), что будет через 30 дней, как перейти на платный тариф. Файл предназначен для копирования на внешний лендинг (не рендерится в этом приложении).

## Открытые вопросы / зависимости

- Пользователь должен завести Resend-аккаунт и `RESEND_API_KEY` в Railway env до реализации email-verification части.
- Нужно решить дизайн письма (шаблон) — минимально текстовый, без брендированного HTML в первой версии.
